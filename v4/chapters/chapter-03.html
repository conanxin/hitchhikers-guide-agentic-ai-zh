第 03 章 · 系统基础

LLM 系统基础：GPU、显存、并行与服务化

从模型算法走向可运行的基础设施

LLM 的系统基础大规模系统架构与基础设施

本章导读

本章把原材料中的相关内容重新组织为学习版讲解，重点追求清晰、连贯和可复习。原始细节可回到 V3 结构化文档或 arXiv PDF 查阅。

学习目标

  • 理解显存为何是 LLM 系统瓶颈
  • 区分数据并行、张量并行和流水线并行
  • 理解服务化中的吞吐、延迟和批处理

核心概念

  • 标记 / tokenToken

    语言模型处理文本的基本离散单位。

  • TransformerTransformer

    以自注意力为核心的序列建模架构。

  • 注意力Attention

    根据查询、键和值计算上下文加权表示的机制。

  • 策略Policy

    在状态下选择动作或生成 token 的概率分布。

  • 缓冲区Buffer

    保存经验、轨迹或中间数据以供训练和分析的结构。

  • 推理Inference

    模型在部署或测试时生成输出的过程。

  • 近端策略优化（PPO）PPO

    通过裁剪目标限制策略更新幅度的强化学习算法。

  • 旋转位置嵌入（RoPE）RoPE

    通过旋转变换注入相对位置信息的位置编码方法。

显存是第一约束

大模型部署时，参数、激活、优化器状态、KV cache 都会占用显存。训练时还要保存梯度和优化器状态；推理时则重点关注权重加载、KV cache 与并发请求。

并行策略怎么选

数据并行适合扩展 batch；张量并行把单层矩阵计算切到多卡；流水线并行把模型层切段。真实系统通常会组合这些方法，再用 ZeRO/FSDP 降低冗余显存。

服务化关注什么

服务化不只是把模型放到 API 后面。调度器、动态批处理、KV cache 管理、限流、监控和故障恢复共同决定用户体验与成本。

关键图表 / 表格：常见并行方式

方式解决什么问题代价
数据并行提高训练吞吐需要同步梯度
张量并行单层计算放不下一张卡跨卡通信频繁
流水线并行模型层数或参数过大有气泡开销和调度复杂度
ZeRO/FSDP减少状态冗余实现和检查点更复杂

公式解释

系统估算常用“显存预算 = 权重 + KV cache + 激活/中间状态 + 框架开销”。V4 用预算框架替代 OCR 破碎公式，便于实际部署时套用。
