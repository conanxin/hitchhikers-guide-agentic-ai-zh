LLM评估

评估是任何严格的机器学习流程的支柱，但它可能是最重要的 在大型语言模型的开发中未被充分认识的组件。与经典监督不同 学习，其中带有真实标签的保留测试集提供了干净的信号，评估LLM 需要解决开放式生成、主观质量判断、多步骤推理等问题 链，以及始终存在的基准测试污染风险。本节提供了系统的 评价景观的处理：从评价类型的分类和评价机制 人工注释，通过排名指标的数学和LLM作为法官的实用性， 到悄然破坏评估流程的陷阱。

为什么LLM的评估很难

LLM 评估与经典 ML 评估之间存在三个基本挑战：

1.输出空间无界。语言模型可以产生任何字符串；很少有一个 单一正确答案。

2. 质量是多维的。乐于助人、真实、安全、连贯和风格是 不同的轴可能会相互权衡。

3.评价本身就是一项语言任务。判断一个响应是否良好需要 理解，这意味着评估容易受到与生成相同的故障模式的影响。

在收集单个数据点之前，从业者必须决定测量什么以及如何测量 它。有原则的分类法可以防止因方便而不是选择指标而犯的常见错误 而不是与部署目标保持一致。

评估类型分类

内在评价与外在评价。 内在评估衡量模型输出的属性 孤立地，不参考下游应用程序。保留语料库上的困惑，BLEU 参考翻译的得分以及编码基准测试的 pass@k 都是内在的。外在的 评估衡量模型对现实世界任务或系统的影响：是否集成 将LLM纳入客户服务渠道可以降低问题升级率吗？是否有编码助手 提高开发速度？

内在与外在的差距

内在指标成本低廉且可重复，但通常与现实世界的效用相关性较差。 困惑度较低的模型不一定更有帮助。外在指标是昂贵的 缓慢但直接衡量我们关心的内容。成熟的评估策略使用内在的 用于快速迭代的指标和用于最终验证的外部指标。

一个有用的决策框架：

• 如果任务有明确的正确答案（数学、代码、事实 QA）：使用精确匹配或执行 - 基于指标。

• 如果任务是开放式的，但有参考输出：使用基于参考的指标作为较低的指标。 绑定，补充LLM作为法官。

• 如果任务是主观的（帮助性、语气、创造力）：使用人工评估或经过良好校准的评估 LLM法官。

• 如果任务涉及多步智能体行为：使用任务成功率和轨迹效率 （第 14.6 节）。

人工注释流水线

强大的注释流水线由五个阶段组成：

1. 任务定义。精确指定注释任务：评级内容、评级范围、 以及以什么标准。此阶段的歧义会传播为嘈杂的标签。

2. 指南制定。使用涵盖边缘的工作示例编写注释指南 案例。在全面部署之前，与一个小型试点小组进行迭代。

3. 标注员招聘和训练。选择具有适当背景的注释者 知识。进行校准会议，其中注释者标记相同的示例并进行讨论 分歧。

4、质量控制。将具有已知标签的黄金标准示例嵌入到注释队列中。 标记黄金示例准确性低于阈值的注释者。

5.聚合。使用多数投票、平均或a组合每个项目的多个注释 概率模型（例如 Dawid-Skene）。

Kappa 的局限性

Kappa 对类别的流行程度很敏感：当一种类别占主导地位时，kappa 可以 即使原始一致性很高（kappa 悖论），也很低。对于序数尺度，加权 kappa （根据分歧的距离按比例惩罚分歧）更合适。对于LLM 评估，评级通常采用 1-5 Likert 量表，始终报告加权 kappa。

注释指南设计

有效的注释指南有几个共同的属性：

• 可操作的标准。用具体的、可观察到的替换诸如“有帮助”之类的模糊术语 行为：“响应直接解决了用户的问题并提供了所有信息 需要完成规定的任务。”

• 工作示例。每个评级级别至少提供两个示例，包括边缘案例。

• 决策树。对于复杂的任务，可以使用流程图来指导注释者完成一系列任务 二元决策减少了认知负荷并提高了一致性。

• 明确的范围。说明注释者不应考虑哪些内容（例如，“不要因风格而受到惩罚”） 偏好；只关注事实的准确性”）。

进化指导

Evol-Instruct [256]通过迭代地重写指令来进化种子指令集 复杂或多样。应用了两个演化算子：

• 深度演化：添加约束、增加推理步骤、具体化抽象、深化 领域知识要求。

• 广度演变：针对相关但不同的主题生成新指令，增加 话题多样性。

如果一条指令通过了消除过滤器，则该指令被接受：演化后的指令不得被 一份简单的副本，不得包含“对不起”或类似的拒绝，并且不得短于 原创。

宪法人工智能数据生成

宪法人工智能（CAI）[129]通过模型批判并修改其模型来生成偏好数据 根据一套原则（“宪法”）自己的产出。流水线：

1.监督学习阶段：对有害提示进行采样，生成初始响应，然后 促使模型根据宪法原则批评响应并对其进行修改。 使用修订后的响应作为监督微调目标。

2. RL 阶段：生成响应对（原始响应与修订后响应），使用模型来标记哪个是 更符合宪法，并在这些标签上训练偏好模型。使用偏好模型作为 RLHF 的奖励信号。

这种方法生成偏好数据，无需人工标记有害内容，从而减少 注释者接触令人痛苦的材料。

评估数据的蒸馏

强大的教师模型（例如 GPT-4）可以生成高质量的评估数据来训练 较小的法官模型。蒸馏流水线：

1. 收集不同的提示和模型响应。

2. 利用老师给出详细的判断（分数+理由）。

3. 在（提示、响应、判断）三元组上微调较小的模型。

4. 根据保留的人工注释验证学生判断。

蒸馏偏差 从单一老师提炼出来的学生评判继承了老师的偏见，包括冗长 偏见（喜欢较长的回答）、自我增强偏见（如果老师也是榜样） 评估）和位置偏差。始终根据独立的人类来验证精炼的法官 注释。

竞技场式成对生成

Chatbot Arena [257]通过众包战斗平台生成评估数据，用户可以在该平台上 提交提示并投票选出他们更喜欢的两个匿名模型回复中的哪一个。这会产生一个 大规模、自然多样化的成对偏好数据集。关键设计选择：

• 匿名化：隐藏模特身份以防止品牌偏见。

• 用户提交的提示：确保提示的多样性和现实世界的相关性。

• 平局处理：用户可以声明平局或表明两个响应都不好。

• 重复数据删除：过滤接近重复的提示，以防止常见的提示过多 查询。

SWE 基准测试方法

SWE-bench [266] 在现实世界的软件工程任务中评估LLM：给定一个 GitHub 问题 描述和存储库代码库，模型必须生成解决问题的补丁。 评估进行如下：

2.模型生成补丁（统一diff格式）。

3. 补丁被应用到存储库。

4. 执行存储库的测试套件；如果所有测试都通过，则任务成功。

主要指标是已解决百分比：生成的补丁通过的问题比例 所有测试。 SWE-bench Verified 是由人工注释者验证的 500 个问题的精选子集 可解且明确。 SWE-bench Lite 是一个包含 300 个问题的子集，旨在加快评估速度。

SWE-bench 关键统计数据（截至 2024 年）

• 完整基准测试测试：来自 12 个流行 Python 存储库的 2,294 个任务。

• 最佳开源智能体：∼43% 已解决（SWE-bench 已验证）。

• 人类表现：∼87% 已解决（每项任务 15 分钟）。

• 评估成本：基于API 的模型每项任务约0.25 美元。

LLM法官表现出立场偏见：对特定领域出现的反应的系统偏好 位置（第一个或最后一个）。这种偏差可能高达 10-15 个百分点。缓解策略：

G-Eval [270] 是基于 LLM 的评估的结构化框架，使用思维链提示 和token概率加权以产生更可靠的分数。框架：

2. 概率加权评分：对于每个评分值 s ∈{1, 2, 3, 4, 5}，得到 来自判断模型的对数概率 log Pθ(s | 提示、步骤、响应)。最终得分为 概率加权平均值：

G-Eval 分数 =

公式
s=1 s · elog Pθ(s) P5 s′=1 elog Pθ(s′) (14.37)

3. 归一化：通过除以最大分数将分数映射到 [0, 1]。

与直接提示相比，G-Eval 与人类判断的相关性更高，特别是对于 诸如连贯性和一致性之类的细微差别，因为概率加权捕获了 判断的不确定性，而不是强迫做出离散的选择。

为什么 G-Eval 有效

标准提示要求法官输出一个标记（例如“4”），这会丢弃模型的 不确定性。 G-Eval 读取所有分数标记的概率分布，有效地计算 法官认为的预期分数。这类似于使用后验平均值 分布而不是模式。

即使精心设计的评估流程也可能产生误导性的结果。本节目录 最常见的故障模式。

基准测试污染

当评估数据出现在模型的训练集中时，就会发生基准测试污染，无论是 直接（逐字包含）或间接（释义或语义相似的内容）。被污染 模型获得的分数过高，无法反映真实的泛化能力。

检测方法：

• n-gram 重叠：计算具有高 n-gram 重叠的评估示例的比例（例如， ROUGE-L > 0.8）与训练语料库。

• 成员资格推理：使用成员资格推理攻击来估计以下概率： 每个评估示例都在训练集中。

• 金丝雀字符串：在评估示例中嵌入唯一的、随机生成的字符串并检查 如果模型可以完成它们。

• 时间保留：使用模型训练截止日期之后创建的评估数据。

缓解措施：

• 维护一个从未公开发布的私有测试集。

• 定期用新示例刷新基准测试。

• 报告训练数据截止日期和净化程序。

与基准测试过度拟合

即使没有直接污染，模型也可以针对特定基准测试进行隐式优化 通过反复评估和超参数调整。这是自适应过度拟合的一种形式： 基准测试测试将信息泄露到模型开发决策中。

基准测试生命周期

随着研究社区对其进行优化，基准测试的效用会随着时间的推移而降低。 MMLU[271]， 曾经是对世界知识的一项具有挑战性的测试，现在模型已经达到了接近人类的性能，但 这些模型在新的知识任务上仍然失败。新基准测试应被视为临时性的 信号源，而不是永久的地面实况。

古德哈特评价定律

古德哈特定律指出：“当一项措施成为目标时，它就不再是一个好的措施。” [272] 在 LLM评价，这体现在几个方面：

• 奖励黑客：使用 RLHF 训练的模型学习利用奖励模型而不是 真正改善。模型可能会学会产生冗长、听起来自信的回应 在奖励模型上得分很高，但实际上是不正确的。

• 度量游戏：经过微调以最大化 BLEU 或 ROUGE 的模型可能会产生以下输出： 在这些指标上得分很高，但对人类来说用处不大。

• 法官博弈：接受LLM法官反馈训练的模型可能会了解法官的偏见 （例如，冗长偏见）而不是真正提高质量。

对古德哈特定律的辩护

1. 指标多样性：使用不同系列的多个指标；一个游戏模型 指标可能不会同时进行。

2.保留评估：维护训练或模型中未使用的评估指标 选择。

3. 人工抽查：定期对模型输出进行抽样以供人工审核，独立于 自动化指标。

4. 对抗性评估：主动探测自动化指标遗漏的故障模式。

5. 外部验证：定期根据外部结果验证内部指标。

额外的陷阱

迅速灵敏。 由于评估的微小变化，LLM的表现可能会发生巨大变化 提示（例如，添加“一步一步思考”或更改答案格式）。始终报告准确的信息 使用提示并考虑对多个提示变体进行评估。

聚合文物。 不同难度级别和分数的任务的平均分数 分布可能会产生误导性的总体指标。一个擅长完成简单任务但失败的模型 在困难任务中可能与具有统一性能的模型具有相同的平均分数。

人类评价中的选择偏差。 人类评估者不是最终用户的随机样本。 众包平台上的注释者可能有不同的偏好、文化背景和 领域知识高于目标用户群体。

评估-部署不匹配。 评估提示通常更短、更清晰、更多 比真实的用户查询格式良好。在基准测试提示上表现良好的模型可能会降级 显着影响生产中发生的嘈杂、模糊、多轮对话。

评估设计的关键问题

在部署评估流水线之前，询问：

1. 评估指标是否与部署目标一致？

2. 评估数据是否代表目标分布？

3. 是否评估了污染和过度拟合风险？

4. 是否报告了所有指标的置信区间？

5. 评估是否可重复（固定种子、版本化提示、公共测试集）？

6. 评估是否根据人类判断或外部结果进行了验证？
