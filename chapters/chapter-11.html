SFT 最佳实践和技术

有监督微调（SFT）是 RLHF 流水线的基础。 SFT 的质量 模型决定了 RL 可以实现的上限：RL 可以细化和改进行为，但是 它无法可靠地引入 SFT 模型中完全不存在的行为。本节 涵盖有效 SFT 的关键技术。

顺序打包以提高效率

填充问题

标准 SFT 批次将所有序列填充到批次中最长序列的长度。对于 具有高长度方差的数据集（例如，短指令和长文档的混合），这 50-80% 的计算浪费在填充token上。顺序打包消除了这种浪费。

序列打包将多个短示例连接成长度为 max_seq_length 的单个序列， 由 EOS token分隔。注意力掩码确保来自不同示例的标记不会 互相照顾：

1. 按长度对示例进行排序（可选，提高打包效率）。

2. 贪婪地将示例打包到大小为 max_seq_length 的 bin 中。

3. 使用块对角注意力掩模来防止交叉示例注意力。

4. 仅计算非填充token的损失。

包装效率

• 典型填充效率：85–95%（带填料时为20–50%）。

• 加速比：对于长度差异较大的数据集，加速2-4 倍。

• 内存：类似于填充（每批的token总数相同）。

• 注意力事项：需要仔细注意力遮盖以避免交叉污染。

为什么聊天模板很重要

语言模型是根据原始文本进行训练的，但指令跟踪模型需要区分 系统提示、用户消息和助理响应之间。聊天模板对此进行编码 结构到token序列中。在推理时使用错误的模板（或没有模板） 导致性能显着下降。

仅完成掩码

为什么要屏蔽提示？

在指令微调中，模型应该学习生成助手的响应，而不是 预测用户的问题或系统提示。提示token上的计算损失浪费 梯度信号，可能导致模型“记住”提示而不是学习响应 给他们。仅完成掩码将所有非辅助标记的损失设置为零。

当 SFT 受伤时——灾难性的遗忘和调整 税

随着LLM通过连续的训练阶段过渡——预训练→继续预训练→ SFT →RLHF/DPO — 性能下降经常出现在标准基准测试测试中。两个 根本上不同的现象驱动了这些回归，混淆它们会导致错误的结果 缓解策略。

灾难性遗忘（结构性擦除）

灾难性遗忘

灾难性遗忘是一种无意的优化失败：当网络优化时 随后在不相交的分布 DB 上训练分布 DA，需要更新权重 对于 DB，物理上覆盖编码 DA 的参数结构：

text
θt+1 = θt −η∇θLB(θt)
=⇒
LA(θt+1) ≫LA(θt)
(10.1)

知识被破坏——编码任务 A 的权重不再存在。这是不可逆转的 无需再训练。

症状：

• 完全分解任务而不是微调数据（例如，模型在完成任务后忘记了如何做数学） 聊天数据上的 SFT）

• 语言多样性的损失——模型仅以微调分布的狭窄方式生成

• 微调期间未强化的知识的事实准确性降低

• 纯英语 SFT 后多语言能力下降

机械原因 — Fisher 信息视角：Fisher 信息矩阵 F 任务 A 确定哪些参数对于 DA 来说“重要”：

公式
F = Ex∼DA h ∇θ log πθ(x) ∇θ log πθ(x)T i (10.2)

具有高 Fisher 特征值的参数对于任务 A 至关重要。 任务 B 完全忽略这些特征值 - Δθ 沿着 ∇LB 点，无论它是否破坏 洛杉矶的高费舍尔指示。

结盟税（行为约束）

对齐税是一种有意的、预期的权衡：模型的原始能力（无约束） 生成，最大推理带宽）减少，因为策略被限制为产生 安全、格式良好、符合偏好的输出。 机制：在DPO/PPO期间，策略πθ因偏离参考πref而受到惩罚 通过 KL 散度：

公式
rimplicit(x, y) = β log πθ(y|x)

πref(y|x) (10.3)

这种束缚限制了模型的输出分布——它无法探索高方差 推理路径与参考偏离太远。知识不会被抹去；它被压制了。 该模型仍然“知道”答案，但其分布趋于安全、通用的响应。 症状：

• 过度拒绝（对于良性询问“我无能为力”）

• 文体僵化——限制性词语、过多的警告、冗长的安全免责声明

• 原始能力基准测试（MMLU、HumanEval）得分较低，同时偏好有所提高 基准测试测试（MT-Bench、AlpacaEval）

• 生成复杂、高熵输出的能力降低（创意写作、新颖算法）

连接到 RL – SFT 质量决定 RL 上限

SFT-RL 关系

SFT 模型是 RL 训练的起点。强化学习可以：

• 放大SFT 模型中存在但较弱的行为。

• 抑制存在但不受欢迎的行为。

• 改进回复的风格和格式。

RL 不能：

• 引入SFT 模型中完全没有的功能。

• 在SFT 阶段从严重的灾难性遗忘中恢复。

• 补偿存在系统性偏差的奖励模型。

SFT 中的探索与利用权衡

为了让强化学习发挥作用，SFT 模型必须偶尔产生正确的响应（因此奖励信号 是非零）。如果 SFT 模型从未对给定的提示产生正确的响应，则 RL 无法 学会产生正确的反应——没有积极的信号可以放大。这就是 SFT 品质的原因 是 RL 性能的上限。 具体来说：如果 SFT 模型正确解决了 10% 的数学问题，强化学习就有可能推动 这个到80%。如果 SFT 模型解决了 0% 的数学问题，RL 将不会取得任何进展（所有奖励 为零，所有优势为零，没有梯度）。

实际意义

1. SFT数据质量：使用高质量、多样化的数据。少量高质量数据 优于大量低质量数据。

2. SFT 数据覆盖范围：确保 SFT 数据覆盖您想要通过 RL 改进的任务。如果一个 任务不在 SFT 数据中，RL 将陷入困境。

3. SFT训练时长：不要过度训练SFT模型。过度训练会降低多样性 并使 RL 探索变得更加困难。

4. 预热：在 RL 之前考虑对特定于任务的数据进行简短的 SFT 预热，即使基础数据 模型已经经过指令调整。

SFT 最佳实践总结

1. 使用序列打包来最大化 GPU 利用率。

2. 应用仅完成掩蔽将梯度聚焦在助理响应上。

3. 使用适合您的车型系列的正确聊天模板。

4. 按比例混合数据与温度缩放 (T ≈2)，以实现多任务 SFT。

5. 使用LoRA防止灾难性遗忘。

6. 在开始 RL 之前评估 pass@k，以确保 SFT 模型是一个可行的起点。

7. 不要过度训练：1-3 个 epoch 通常足以进行指令微调。

8. 监控多样性指标（熵、n-gram 多样性）以检测模式崩溃。
