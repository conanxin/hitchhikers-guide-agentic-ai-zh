What Is a Skill?

原 PDF p.412

智能体技巧

随着智能体从单一的提示和工具系统发展到模块化架构，一个关键的设计 问题出现了：智能体的能力应该如何组织、发现和组合？的 答案越来越集中在技能的概念上——离散的、可重复使用的行为单元 无需重新训练即可加载、组合和交换。 这个想法被 Voyager [228] 所推广，它证明了 Minecraft 中的 LLM 智能体 可以积累一个不断增长的可执行代码技能库，每个技能都经过验证和存储以供以后重用。 同样的原则也适用于生产智能体：技能将领域专业知识封装在可组合的、 可版本控制的格式超出了任何单个提示可以容纳的范围。技能经常包裹MCP 用于工具访问的服务器（第 21 章），将技能抽象连接到标准化工具层。

什么是技能？

技能是一个独立的能力模块，为客服人员提供特定领域或领域的专业知识 任务。与原始工具（公开单个功能）不同，技能包括：

• 系统提示增强：特定领域的指令、约束和角色 元素注入到智能体的上下文中。

• 工具绑定：技能所需的一个或多个工具（API、MCP 服务器、本地命令）。

• 知识：智能体需要的参考材料、示例或少量演示 正确执行技能。

• 工作流logits：指导工作的多步骤过程、决策树或条件流 智能体完成复杂的任务。

• 护栏：技能特定的安全约束、输出格式要求和验证规则。

技能 vs. 工具 vs. 智能体

概念 适用范围 示例

工具 单一函数调用 网络搜索（查询） 技能 相干能力（提示 + 工具 + 知识） “研究分析师”技能

智能体 具有多个自治实体 技巧 编码助理

工具是锤子。一项技能是知道如何建造房屋。智能体是木匠，他 选择要应用的技能。

技能架构模式

静态技能加载

最简单的模式：根据配置在智能体初始化时加载技能。智能体 始终可以使用其所有技能。

# Pseudocode
-- framework -agnostic
pattern
agent = Agent(

model="claude -sonnet -4 -20250514",
skills =["code -review", "documentation ", "testing"],
# Each
skill
adds prompts , tools , and
knowledge to the agent
)

优点：简单、可预测、低延迟。

缺点：未使用技能时上下文窗口会浪费；无法扩展到数百种技能。

动态技能发现

智能体根据当前任务选择激活哪些技能。技能路由器（通常是轻量级的 分类器或基于嵌入的匹配器）确定相关性：

# Pseudocode
-- framework -agnostic
pattern
relevant_skills = skill_router.match(

user_request=message ,
available_skills =skill_registry ,
max_skills =3
)
agent.activate( relevant_skills )

优点：可扩展到大型技能库；上下文有效。

缺点：路由错误可能会错过相关技能；增加延迟。

层次技能构成

技能可以依赖于其他技能，形成 DAG。高级技能（例如“部署应用程序”）可能 调用子技能（“运行测试”、“构建 Docker 映像”、“更新 DNS”）：

• 技能明确声明依赖关系

• 编排器在执行前解析依赖关系图

• 子技能可以在多个父技能之间共享

案例研究：Anthropic 的智能体设计

Anthropic 的智能体架构方法 [342] 提供了最清晰的技能阐述之一—— 生产中基于智能体设计。他们的哲学强调简单而不是复杂 整体框架上的可组合构建块。 （这些模式也涵盖 从第 19 章的编排角度来看。）

核心原则

  1. 从最简单的解决方案开始。在找到更简单的方法之前不要寻求智能体模式 （单一LLM调用，检索+生成）已被尝试并发现不足。

强化LLM

在 Anthropic 的框架中，基本单位不是裸模型，而是增强的 LLM：

增强LLM = 模型 + 检索 + 工具 + 记忆

这直接映射到技能概念：每个技能配置哪些检索源、工具和 内存存储模型可以访问的特定任务。技能边界定义了模型的内容 可以在特定调用中查看并执行操作。

Skill Registries and Marketplaces

原 PDF p.415

智能来自（1）模型的能力，（2）工具描述的质量，以及 (3) 任务框架的清晰度——不是来自复杂的编排logits。技能提供 (2) 和 (3) 的结构。

Anthropic 方法的设计建议：

• 保持智能体循环简单：避免过度设计控制流。让模型决定吧。

• 投资工具质量：详细、明确的工具描述比工具更有价值 复杂的路由logits。

• 使用结构化输出：强制模型以可解析格式（JSON、 函数调用）——减少技能执行错误。

• 内置恢复：技能应该优雅地处理错误——使用不同的参数重试， 要求澄清，或升级给人工。

• 限制每项技能的范围：试图做所有事情的技能将一事无成。狭窄， 明确的技能比广泛的技能更有效。

技能生命周期

  1. 发现：系统识别哪些技能可用（注册、市场、本地 定义）。

2.选择：根据用户需求，匹配并加载相关技能。

  1. 激活：技能提示、工具和知识被注入到智能体的上下文中。

4.执行：智能体使用技能的能力来完成任务。

  1. 停用：技能上下文被删除，以释放上下文窗口空间以用于后续任务。

6.学习：执行结果可能会更新技能的少数样本或微调路由。

技能注册和市场

生产技能系统需要基础设施：

• 技能清单：结构化描述（名称、能力、所需工具、输入/输出 schema）实现自动发现和路由。

• 版本控制：技能不断发展；智能体需要固定特定版本以实现可重复性。

• 依赖性解析：技能可能需要特定的MCP 服务器、API 密钥或其他技能。

• 权限模型：并非所有智能体都应该有权访问所有技能（安全性、成本、能力） 边界）。

• 市场：组织可以发布、共享和安装技能——类似于软件包 代码管理器。
