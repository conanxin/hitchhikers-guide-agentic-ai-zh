智能体 UI 框架

随着大型语言模型从被动文本生成器转变为能够规划的主动智能体， 工具的使用和多步骤推理，人类与它们交互的界面必须 并行发展。传统聊天界面——专为单轮或短上下文对话而设计—— 不适合智能体工作流程的需求：长时间运行的任务、分支决策树、 并行工具调用，以及有意义的人类监督的需要。本节调查了 智能体 UI 框架的景观：设计范例、组件库和实现 能够实现丰富、透明且值得信赖的人机协作的模式。

为什么智能体需要专门的接口

聊天气泡传达结果。智能体 UI 传达一个过程——推理、调用的工具、 做出的决定以及需要人类判断的点。如果没有这种可见性， 用户无法信任、纠正或向智能体学习。

聊天界面和智能体界面之间的差距反映了自动售货机之间的差距 机器和熟练的合作者。当智能体执行 20 步研究任务、浏览网页时， 编写并运行代码，并综合报告，用户需要回答一些简单的问题 文本回复无法提供：

• 座席现在在做什么？长时间运行的任务需要进度反馈；沉默 滋生不信任。

• 智能体为何做出此决定？推理的透明度使用户能够 及早发现错误。

• 使用了哪些工具以及有哪些输入？工具来源对于验证至关重要 事实主张和审计行为。

• 我应该在哪里进行干预？智能体必须浮出决策点以保证人类 判断，而不是让每个微观决策压倒用户。

• 我可以撤消此操作吗？不可逆的操作（发送电子邮件、修改文件、执行代码）需要 显式确认和回滚路径。

自动化偏差风险 对人机交互的研究一致表明，用户过度信任自动化 系统，尤其是当这些系统自信且没有不确定性地呈现输出时 信号[398]。智能体 UI 必须通过暴露不确定性来积极抵消自动化偏差， 展示推理，并使质疑或推翻智能体决策变得容易。

智能体 UI 范式

本节聚焦“智能体 UI 范式”的定义、机制与工程含义。

智能体Agent

能够感知状态、规划行动、调用工具并迭代完成任务的系统单元。

标记 / tokenToken

语言模型处理文本的基本离散单位。

1. 透明性：使智能体的内部状态对用户来说是清晰可见的。

2. 控制：提供有意义的干预点，无需持续监督。

3. 信任校准：帮助用户建立准确的座席能力和心理模型 限制。

4. 效率：最小化认知负荷；在正确的时间呈现正确的信息。

5. 可恢复性：使错误易于检测和逆转。

智能体的 UI 范式

没有单一的 UI 范例适合所有智能体用例。适当的接口取决于任务持续时间， 需要人工参与、输出类型和用户专业知识。频谱范围从完全 对话式聊天界面与完全自主的仪表板相结合，只需最少的人机交互。

基于聊天的界面

聊天模式——消息气泡、文本输入和滚动历史记录——仍然是最熟悉的 LLM互动的入口点。它的优点是学习曲线低和自然语言灵活性。 对于智能体使用，聊天界面增加了：

• 流式响应：token在生成时出现，提供即时反馈 并减少感知延迟。通过服务器发送事件 (SSE) 或 WebSocket 实现。

• 内联工具指示器：消息流中显示的小徽章或可扩展部分 当调用工具时（例如，“[在网络上搜索： 2024 年气候变化]”）。

• 输入指示器和状态消息：“智能体正在思考……”、“正在运行Python 代码……”、 “正在获取结果……”在延迟间隙让用户了解情况。

• 消息线程：对于多轮智能体任务，可折叠子线程可以包含内部消息。 在不扰乱主要对话的情况下调解步骤。

客服人员的聊天 UI 限制

聊天接口将本质上并行的进程串行化。当智能体分散到五个工具时 同时，线性消息流歪曲了实际的执行图。对于复杂的 智能体工作流程中，聊天应该用更丰富的范例来增强或取代。

基于画布和工件的界面

由 Claude Artifacts1 和 ChatGPT Canvas2 推广的画布范式引入了一种分裂 窗格布局：左窗格承载对话，而右窗格（“画布”或“工件”） 面板”）将生成的内容（代码、文档、图表、电子表格）显示为实时、可编辑的 神器。 主要特点：

• 持久工件：生成的内容在各个轮次中持续存在，并且可以迭代地细化 通过自然语言指令（“将图表设为蓝色”、“将错误处理添加到 函数”）。

• 就地编辑：用户可以直接编辑工件，智能体可以观察和响应 到那些编辑。

• 版本历史记录：工件维护修订历史记录，支持回滚到任何先前状态。

• 多工件工作区：高级实施支持多个同时 工件（例如，代码文件、其测试套件和文档页面）。

画布范例特别适合共同创作任务：写作、编码、数据分析、 和设计，其中输出是文档或工件，而不是对话答案。

工作流程可视化

对于执行结构化计划（步骤序列或图表）的智能体，工作流可视化 UI 使计划明确且可追踪。这种范式常见于：

• 智能体流水线（LangGraph、AutoGen、CrewAI）：渲染智能体的执行图 作为有向无环图 (DAG) 或流程图，其中节点代表步骤和边 表示数据流或控制流。

• 任务分解视图：智能体的高级计划以清单或甘特图形式显示 时间轴，每个子任务都会扩展以显示其自己的步骤。

• 实时进度跟踪：节点在执行时改变颜色或显示旋转器；已完成 节点显示输出；失败的节点显示错误详细信息。

LangGraph Studio3 是这种范例的典型示例，提供了基于图形的调试器 和 LangGraph 智能体的可视化工具。用户可以检查每个节点的状态、重放执行情况、 并注入修改后的状态来测试替代路径。

仪表板和监控界面

对于长期运行或生产智能体，仪表板 UI 提供操作视图：

• 实时状态：哪些智能体正在运行、空闲或失败；当前的任务和步骤。

• 资源指标：token消耗、API 调用计数、延迟直方图、成本估算。

• 队列管理：待处理任务、优先级排序、速率限制状态。

• 警报和异常检测：异常行为（过度重试、成本激增、重复 失败）作为通知出现。

• 历史分析：任务完成率、平均持续时间、一段时间内的错误频率。

仪表板 UI 通常使用 Grafana、4 个自定义 React 仪表板或 Streamlit 等工具构建， 并且针对的是运营商而不是最终用户。

协作接口

协作 UI 将智能体视为共享工作空间（文档、代码库、 或与人类合作者一起设计画布。主要特点包括：

• 状态指示符：座席在共享工作区中显示为命名光标或头像。

• 更改归因：座席所做的编辑在视觉上与人工编辑有区别 （例如，颜色编码的差异）。

• 内联建议：智能体以跟踪编辑或评论的形式提出更改建议，由人类 可以接受、拒绝或修改。

• 冲突解决：当智能体和人工同时编辑同一区域时， UI 使冲突浮出水面并促进解决。

这种范例正在 Cursor5（人工智能协作代码编辑）、Notion AI 等工具中出现，6 和 Google Docs 与 Gemini 集成。7

具有检查点的自主性

在自治范围的远端，一些智能体基本上独立运行——浏览 网络、编写代码、执行命令——并且仅在需要人工的预定义检查点出现 批准。该范例用于：

• 计算机使用智能体（Anthropic Computer Use，8 OpenAI Operator9）：智能体控制 浏览器或桌面；用户界面显示实时屏幕提要，并在不可逆转之前暂停以供批准 行动。

• 带门的自动化流水线：智能体完成 CI/CD 式工作流程 阶段并等待人类“合并”，然后再继续。

• 计划智能体：按计划运行并异步报告结果的智能体，具有 基于通知的 UI，用于审查输出并批准后续操作。

Checkpoint UI 实践

负责“清理我的电子邮件收件箱”的智能体可能会自动对 500 封电子邮件进行分类和存档 电子邮件，然后暂停并呈现摘要：“我发现 23 封电子邮件似乎来自邮寄 您已经 6 个月没有打开过的列表。我应该取消全部、部分还是全部取消订阅？”用户 检查列表、做出选择，然后智能体继续进行。这种模式——自主执行 由人类决策点打断——平衡效率与控制。

智能体的关键 UI 组件

无论总体范例如何，智能体 UI 都共享一组重复出现的组件。本节 列出了最重要的目录，并为每个目录提供了设计指南。

思维与推理展示

现代LLM，特别是那些受过思想链或扩展思维训练的LLM（例如 OpenAI o1/o3，具有扩展思维的人择克劳德），之前产生大量的内部推理 产生最终响应。这种推理是一把双刃剑：它增加了透明度 但可能会因为冗长的内心独白而让用户不知所措。 最佳实践：

• 可折叠推理块：显示摘要（“思考 12 秒”）并展开 为需要详细信息的用户切换。

• 渐进式披露：默认只显示最终结论；推理可用于 需求。

• 结构化推理：如果模型产生结构化思维（假设、证据、 结论），用视觉层次结构而不是文本墙来呈现它们。

• 推理与响应的区别：从视觉上清楚地区分内部推理（其中 最终响应中可能包含错误或错误的开始）。

工具使用可视化

工具调用是智能体与世界交互的主要机制。将它们可视化是 对于信任和调试至关重要。

工具调用剖析

每个工具调用都有四个值得显示的组件：(1) 工具名称和图标，(2) 输入参数（可能很大的 JSON），(3) 输出/结果（可能很大），以及 (4) 计时（延迟）。 UI 必须平衡完整性和可读性。

工具可视化的设计模式：

• 内联工具卡：消息流中的紧凑卡显示工具名称、一行 输入摘要和状态（运行/成功/错误）。可扩展以获取完整详细信息。

• 工具时间线：水平时间线依次显示所有工具调用以及持续时间，使 识别瓶颈。

• 输入/输出差异：对于修改状态（例如文件编辑）的工具，显示之前/之后的差异。

• 工具图标和品牌：常用工具的可识别图标（网络搜索、代码执行、 文件系统、API）可实现快速扫描。

• 错误突出显示：失败的工具调用以红色显示，并带有错误消息和任何重试 尝试。

进度指标

多步骤智能体任务需要丰富的进度反馈：

• 步骤级进度：计划步骤的编号列表，每个步骤完成时都带有复选标记。 对于动态计划，可以根据智能体的调整添加或删除步骤。

• token流指示器：生成期间闪烁的光标或动画省略号；一个 高级用户的每秒token计数器。

• 预计完成时间：在可行的情况下，根据任务复杂性和历史情况制定预计完成时间 性能。以适当的不确定性显示（“大约 2-5 分钟”）。

• 子任务嵌套：对于分层任务，具有可扩展的树形结构进度视图 子任务。

• 取消：清晰可见的“停止”按钮，可以优雅地停止智能体并进行总结 至此工作已完成。

审批门

批准门是人在环控制的主要机制。它们必须被设计 提供丰富的信息（为用户提供足够的背景信息以做出正确的决定）而不感到疲劳 （每一个小动作都需要批准）。

批准门疲劳警报

如果智能体过于频繁地请求批准，用户将开始本能地批准而不阅读 - 违背了门的目的。分级审批政策（参见第 26.7 节）对于 保持有意义的监督。

审批门 UI 元素：

• 行动摘要：以通俗易懂的语言描述客服人员想要执行的操作（“发送电子邮件 发送至 john@example.com 并附上报告”）。

• 风险指示器：行动可逆性的视觉信号（绿色=容易撤消，黄色=很难撤消） 撤消，红色 = 不可逆转）。

• 批准/拒绝/修改：三选项界面； “修改”打开一个编辑器 批准前的操作参数。

• 上下文面板：可展开部分显示客服人员想要采取此操作的原因（相关 推理、先前步骤）。

• 超时行为：明确指示如果用户不响应（智能体 暂停，不继续）。

上下文显示

智能体维护内部状态——内存、活动工具、检索到的文档、对话历史记录—— 这会影响他们的行为。使这种状态可见有助于用户理解和预测智能体 行为。

• 内存面板：显示智能体当前“记住”的有关用户、任务和之前的内容 互动。由用户可编辑。

• 活动工具列表：智能体当前可以使用哪些工具，以及启用/禁用切换。

• 检索到的上下文：当前位于智能体上下文窗口中的文档或数据块，其中 来源引用。

• token预算指标：消耗了多少上下文窗口，帮助用户 了解何时开始新会话。

错误和恢复用户界面

智能体失败——工具返回错误，模型产生幻觉，计划变得不可行。 UI 必须处理 优雅地失败：

• 错误卡：内联显示故障以及错误类型、消息和智能体的解释 状态。

• 重试控制：带有可选参数调整的手动重试按钮。

• 替代方法：当主要方法失败时，智能体会提出替代方案； 用户界面将它们呈现为可选择的选项。

• 部分结果：如果多步骤任务中途失败，UI 会显示已完成的步骤及其结果 输出，保留部分值。

• 升级路径：当智能体出现问题时，提供人工支持或手动完成的清晰路径 无法继续。

信心指标

LLM是具有校准（或错误校准）不确定性的概率系统。展现自信 帮助用户知道何时信任以及何时验证：

• 口头对冲显示：突出显示诸如“我不确定”或“您可能想要验证”之类的短语 提请注意力低可信度的主张。

• 来源质量指标：对于检索到的信息，显示来源的新近度、权威性和 相关性分数。

• 明确的不确定性请求：“您有多大信心？”提示智能体的按钮 自我评估并解释其不确定性。

• 验证建议：对于高风险输出，智能体主动建议验证 步骤（“我建议独立检查此计算”）。

清单 26.3：使用 FastAPI 进行 SSE token流式传输

工具调用流

现代 LLM API 支持流式工具调用：工具名称和参数被流式传输 逐渐地，使 UI 能够显示“智能体正在调用 search_web 并查询：‘气候变化 2024 年’。。。在调用该工具之前。这需要解析部分JSON，可以是 使用流式 JSON 解析器完成。 工具调用流的模式：

• 渐进式参数显示：在工具参数流入时显示工具参数，甚至在调用之前显示 已完成。

• 并行工具调用指示器：当模型同时调用多个工具时，显示所有工具 其中一些处于待处理状态，然后在结果到达时更新每个。

• 工具结果流式传输：某些工具（例如代码执行、网页抓取）本身可以流式传输 结果；将这些逐渐通过流水线传送到 UI。

多智能体流

在多智能体系统中，多个智能体可能同时生成输出。用户界面必须 处理并行流：

• 智能体标记的流：每个流都标有智能体的身份； UI 呈现 它们位于单独的车道或面板中。

• 流合并：对于主管-子智能体模式，主管的流可能会交错 具有子智能体流； UI 必须保持一致的顺序。

• 背压：如果 UI 无法像流到达一样快地呈现（例如，多个智能体生成 同时），反压机制必须防止缓冲区溢出。策略包括： 删除中间token（仅显示最新的）、批量更新或暂停较慢 溪流。

• 当用户发送消息时，该消息会立即出现在聊天历史记录中（乐观地），同时 该请求正在处理中。

在高吞吐量智能体场景中，传入数据的速率可能会超过 UI 的渲染能力。 管理背压的策略：

• 节流更新：对于指标和状态显示，以固定速率（例如 10 Hz）更新 无论输入数据速率如何。

何时中断智能体

并非所有智能体行为都需要人工审查。原则性中断政策考虑：

• 可逆性：始终不可逆的操作（删除文件、发送电子邮件、购买） 授权批准。可逆操作（读取文件、搜索网络）通常不会。

• 范围：影响外部系统或其他人的行为比纯粹的行为需要更多的审查 本地行动。

• 置信度：当智能体对用户意图的解释的置信度较低时，它会 应该要求澄清而不是继续。

• 成本：高成本的操作（大量 API 调用、昂贵的计算）需要批准。

• 新颖性：在这种情况下，智能体之前没有采取过的行动值得比之前更严格的审查 日常动作。

分级审批流程

分级审批政策兼顾监管与效率：

三级审批模式

第 1 层（自动批准）：低风险、可逆、常规操作。示例：网络搜索、阅读 文件，调用只读 API。智能体不间断地继续进行；记录操作以供审核​​。 第 2 层（通知）：中等风险操作。 UI 显示非阻塞通知（“智能体发送了 草稿电子邮件到您的草稿文件夹”），用户可以异步审阅。一个简短的窗口（例如， 30 秒）允许在操作完成之前取消。 第 3 级（需要批准）：高风险、不可逆转或高成本的行动。智能体暂停并 提出了一个阻塞审批门。用户必须明确批准、拒绝或修改之前 智能体继续。

用户可以配置层之间的阈值（“发送电子邮件之前始终询问”） 或从用户行为中学习（如果用户始终批准网络搜索，则在 未来）。

可达性和信任

信任不是一个特征——它是一个系统的新兴属性，它始终如预期地运行， 清晰地解释自己，并从失败中优雅地恢复。智能体 UI 的设计必须考虑到信任 作为一流的关注点。

解释智能体的决定

智能体 UI 中的可解释性不仅仅是显示思维链。它需要：

• 决策理由：对于后续决策，智能体不仅应该解释其内容 决定了，但为什么——考虑了哪些因素，拒绝了哪些替代方案，以及 做出了假设。

• 来源归属：声明应与其来源相关联；检索到的文档应该是 可引用的。

• 反事实解释：“如果你说 X 而不是 Y，我就会做 Z”——帮助 用户了解智能体的决策边界。

• 不确定性量化：明确的置信度陈述以及驱动因素 不确定性。

显示置信度

置信度指标必须经过校准且有意义：

• 言语自信：自然语言表达（“我相当有信心”、“我不确定 对于大多数用户来说，这个”）比数字概率更容易解释。

• 视觉置信度：颜色编码（绿色/黄色/红色）、图标变体或字体粗细可以编码 无需添加文本即可获得信心。

• 按声明的置信度：对于多声明响应，每个声明的置信度指标（例如，内联 脚注）比单个响应级别分数提供更多信息。

撤消和回滚功能

在技术上可行的情况下，每个相应的智能体操作都应该是可撤销的：

• 带撤消功能的操作日志：所有座席操作的时间顺序日志，带有“撤消”按钮 每个可逆动作。

• 基于快照的回滚：对于有状态任务（例如，代码编辑、文档编写），定期 快照允许回滚到任何先前的状态。

• 试运行模式：在执行计划之前，智能体可以对其进行模拟并显示预测结果 状态更改，允许用户在采取任何实际操作之前批准或修改。

• 优雅降级：当无法撤消时（例如，已发送电子邮件），UI 清楚地传达这一点并提供最佳的可用替代方案（例如，发送后续信息）。

UI 中的审计跟踪

对于企业和受监管的用例，审计跟踪至关重要：

• 不可变的操作日志：每个智能体操作、工具调用和人工批准都记录在案 时间戳、用户身份和完整参数。

• 可导出的历史记录：审计跟踪可以导出为 JSON、CSV 或 PDF 格式以确保合规性 报告。

• 差异视图：对于文档或代码修改，审计跟踪包括之前/之后的差异。

• 会话重播：能够逐步重播整个智能体会话，以进行调试或 合规审查。

管理用户期望

错误的期望是用户不信任的主要原因。智能体 UI 应主动管理 期望：

• 能力披露：清晰、易于理解的文档，说明智能体可以做什么和不能做什么。

• 限制确认：当智能体遇到超出其能力范围的任务时，它会 明确地说出来，而不是默默地尝试和失败。

• 不确定性沟通：主动沟通不确定性，而不是等待 以便用户发现错误。

• 一致的角色：一致的座席身份和沟通风格可增强熟悉度 和可预测性。

总结

Agentic UI 框架代表了人机交互的新领域，需要 从第一原则重新思考界面设计。本节的主要见解是：

1. 范式选择很重要：适当的 UI 范式（聊天、画布、工作流程、dash- 董事会、协作、自主）取决于任务结构、所需的人员参与、 和输出类型。大多数生产系统结合了多种范例。

2. 透明度是不容谈判的：用户不能相信他们看不到的东西。思想展示、 工具可视化和上下文面板不是可选功能 - 它们是基础 值得信赖的智能体系统。

3. 流媒体是基线：用户希望看到智能体实时工作。token流， 工具调用流和多智能体流是赌注功能。

4. 审批门必须分层：统一审批政策（全部审批或全部审批） 什么都没有）在实践中失败。自动批准安全操作并阻止危险的分层策略 保持监督而不造成瓶颈。

5. 生成式 UI 是前沿：LLM 不仅能够生成文本，还能生成 UI 组件——图表、表单、地图、小部件——使界面能够适应内容而不是 而不是将内容强行放入固定模板中。

6. 通过一致性和可恢复性赢得信任：撤消功能、审计跟踪、 校准后的置信度指标与建立用户信任的原始能力同样重要。

设计原则：智能体作为透明的合作者

智能体 UI 设计的北极星是透明的协作者：一个智能体，其行为是 总是可见的，其推理总是可理解的，其错误总是可以纠正的，并且 他们的能力和局限性总是很清楚的。每个 UI 决策都应根据 这个标准。

本节中描述的框架和模式 - Vercel AI SDK、Chainlit、Gradio、Streamlit、 LangGraph Studio—提供构建块。从业者面临的挑战是将它们结合起来 深思熟虑，以用户的特定需求和领域的特定风险为指导。
